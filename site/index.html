Why it's built this way

The decisions behind it

Anyone can run a container. What I wanted to learn was the reasoning — the tradeoffs you only meet when you actually operate something.

01 One service public, everything else private

Exposing a home server usually means forwarding a router port — which reveals your home IP and often can't work at all behind the carrier-grade NAT most Indian ISPs use. Instead, a Cloudflare Tunnel makes an outbound connection from the Pi, so nothing is ever opened inward.

The public path serves exactly one hostname; anything else is refused. Gitea, the password manager, and every dashboard stay reachable only over a WireGuard mesh VPN — so I can reach them from anywhere without any of them touching the public internet.

02 Monitored, not just running

Prometheus scrapes the host and every container every 30 seconds; Loki collects their logs so I can search logs and metrics together in Grafana. The part I found most valuable wasn't collecting data — it was deciding what "unhealthy" means.

I wrote a handful of alert rules as code — service down, low disk, memory pressure, high CPU temperature — and wired Alertmanager to my phone. Stopping a container now sends a Telegram alert within two minutes, and a recovery notice when it comes back.

03 Reproducible from a bare OS

Every config lives in Git from the first commit. An Ansible playbook takes a fresh Ubuntu install to the full running stack in one command — I've actually wiped the Pi and rebuilt it this way, which is what made me trust it.

A GitHub Actions pipeline validates every Compose file and lints the YAML on each push, so a broken config is caught before it ever reaches the Pi.

04 Backups I've actually restored

The data that can't be recovered from Git — Git repositories, the password vault, dashboards, tunnel credentials — is backed up nightly with restic on a systemd timer, encrypted and deduplicated.

A backup you've never restored is only a hope. I ran the restore into a scratch directory and confirmed the files came back readable, and wrote a recovery runbook alongside it.

Being straight about it

What I'd change if I rebuilt it

It runs on a single node, so the Pi itself is a single point of failure. Backups are automated and tested, but there's no live redundancy — if the hardware dies, there's a rebuild-and-restore gap.

I looked at moving a workload onto Kubernetes (K3s) for the experience, measured its real cost — the control plane alone used over a gigabyte of RAM on a 4 GB board — and decided against keeping it, because nothing in the homelab yet needs orchestration. I'd rather add it when there's a genuine reason than run it for the keyword.

Next, I'd add a second node for redundancy and move backups fully off-site. I mention all of this because knowing a system's limits is part of understanding it.
